# Цель занятия
Освоить базовую организацию мониторинга backend-приложения на Go с использованием Prometheus для сбора метрик и Grafana для их визуализации.

## Архитектура

| Сервис | Протокол | Порт | Назначение |
|--------|----------|------|------------|
| Auth Service | gRPC | 50051 | Проверка токенов |
| Tasks Service | HTTP | 8090 | CRUD задач, экспорт метрик |
| Prometheus | HTTP | 9090 | Сбор и хранение метрик |
| Grafana | HTTP | 3000 | Визуализация метрик |

## Метрики приложения

| Метрика | Тип | Лейблы | Описание |
|---------|-----|--------|----------|
| `app_http_requests_total` | Counter | method, path | Общее число HTTP запросов |
| `app_http_errors_total` | Counter | method, path, status_code | Число ошибок |
| `app_http_request_duration_seconds` | Histogram | method, path | Длительность запросов |
| `app_task_requests_total` | Counter | operation, status | Операции с задачами |
| `app_task_request_duration_seconds` | Histogram | operation | Длительность операций |

## Контрольные вопросы
1. **Что такое метрики приложения?** Метрики приложения — это числовые показатели, которые описывают состояние и поведение системы в реальном времени, такие как количество запросов, число ошибок или время обработки. Они представляют собой агрегированные данные, которые позволяют наблюдать за системой в целом, а не за отдельными событиями.

2. **Чем метрики отличаются от логов?** Логи отвечают на вопрос «что именно произошло» и описывают конкретные дискретные события, такие как ошибка при обработке конкретного запроса. Метрики же отвечают на вопрос «как система ведёт себя в целом» и представляют собой агрегированные числовые данные за промежуток времени, например долю ошибок или среднюю задержку. Логи нужны для детального разбора инцидентов, а метрики — для оперативного обнаружения проблем и наблюдения за динамикой системы.

3. **Какую роль выполняет Prometheus?** Prometheus — это система мониторинга, которая регулярно опрашивает HTTP-эндпоинты приложений, собирает метрики и сохраняет их как временные ряды в своей базе данных. Он не принимает метрики по инициативе приложения, а сам «вытягивает» их по расписанию, что упрощает настройку и обнаружение проблем с доставкой метрик. Также Prometheus предоставляет язык запросов PromQL для анализа накопленных данных и интеграции с системами визуализации, такими как Grafana.

4. **Что такое scraping в Prometheus?** Scraping — это процесс периодического опроса HTTP-эндпоинтов приложений для сбора метрик, который выполняется Prometheus согласно настройкам в конфигурационном файле. Prometheus отправляет HTTP-запросы на указанные цели (targets) по определённому интервалу, например каждые 5 секунд, и извлекает метрики в текстовом формате. Такой pull-подход упрощает обнаружение отказавших компонентов и централизованное управление сбором данных.

5. **Зачем приложению маршрут /metrics?** Маршрут /metrics нужен для публикации метрик приложения в формате, понятном Prometheus, чтобы система мониторинга могла их собирать. Этот эндпоинт обычно возвращает текстовое представление всех зарегистрированных счётчиков, гистограмм и других метрик. Без такого маршрута Prometheus не сможет получить данные о состоянии приложения, а мониторинг станет невозможным.

6. **Что делает promhttp.Handler()?** promhttp.Handler() возвращает HTTP-обработчик (http.Handler), который публикует все зарегистрированные метрики в формате, ожидаемом Prometheus. Этот обработчик автоматически собирает метрики из стандартного реестра Prometheus и преобразует их в текстовое представление для отдачи по запросу. Его достаточно подключить к маршруту /metrics, чтобы приложение начало экспортировать метрики.

7. **Для чего нужна Grafana?** Grafana — это система визуализации данных, которая подключается к различным источникам, таким как Prometheus, и позволяет строить графики, дашборды и панели для анализа метрик. Она не собирает данные самостоятельно, а отображает уже накопленные временные ряды в понятном и наглядном виде. Благодаря Grafana разработчик или администратор может в реальном времени видеть динамику нагрузки, ошибок и производительности приложения.

8. **Какие три основные метрики реализованы в этой работе?** В работе реализованы три основные метрики: счётчик общего числа HTTP-запросов (app_http_requests_total), счётчик ошибок с разбивкой по кодам ответа (app_http_errors_total) и гистограмма длительности обработки запросов (app_http_request_duration_seconds). Эти метрики позволяют отслеживать нагрузку на сервис, процент ошибок и скорость ответа. Дополнительно добавлены метрики по операциям с задачами для более детального анализа бизнес-логики.

9. **Что показывает Histogram?** Гистограмма (Histogram) показывает распределение значений измеряемой величины по заданным интервалам, называемым корзинами, что позволяет оценить не только среднее значение, но и процентили. Например, гистограмма длительности запросов показывает, сколько запросов уложилось в 0.1 секунды, сколько в 0.5 секунды и так далее. Это помогает выявлять медленные запросы и оценивать качество обслуживания, так как среднее значение может скрывать наличие долгих выбросов.

10. **Почему мониторинг важен для сопровождения backend-приложений?** Мониторинг позволяет оперативно обнаруживать проблемы, такие как рост числа ошибок или увеличение времени ответа, ещё до того, как на них пожалуются пользователи. Он даёт возможность видеть динамику нагрузки на сервис и планировать масштабирование на основе реальных данных. Без мониторинга разработчик работает вслепую и не может объективно оценить, как изменения в коде влияют на производительность и стабильность системы.
